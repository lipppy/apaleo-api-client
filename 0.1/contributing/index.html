    Contributing to Apaleo API Client\n\nWe welcome contributions to the Apaleo API Client! Whether you're fixing bugs, adding features, improving documentation, or helping with testing, your contributions are valued.\n\n## \ud83d\ude80 Quick Start for Contributors\n\n### 1. Set Up Development Environment\n\nbash\n# Fork the repository on GitHub first, then clone\ngit clone https://github.com/YOUR-USERNAME/apaleo-api-client-python.git\ncd apaleo-api-client-python\n\n# Install with all development dependencies\npoetry install --with dev,test,lint,docs\n\n# Activate the virtual environment\npoetry shell\n\n# Verify installation\ninv help\n\n\n### 2. Check Everything Works\n\nbash\n# Run tests\ninv test\n\n# Run linting\ninv lint\n\n# Format code\ninv format\n\n# Build documentation\ninv docs\n\n\n### 3. Make Your Changes\n\nbash\n# Create a feature branch\ngit checkout -b feature/your-feature-name\n\n# Make your changes\n# ...\n\n# Run tests and linting\ninv test\ninv lint\n\n# Commit your changes\ngit add .\ngit commit -m \"feat: add your feature description\"\n\n# Push to your fork\ngit push origin feature/your-feature-name\n\n\n### 4. Submit a Pull Request\n\n1. Go to the GitHub repository\n2. Click \"New Pull Request\"\n3. Select your branch\n4. Fill out the PR template\n5. Wait for review feedback\n\n## \ud83d\udcc4 Project Structure\n\n Understanding the codebase structure:\n\n\nsrc/apaleoapi/\n\u251c\u2500\u2500 __init__.py # Package exports\n\u251c\u2500\u2500 client.py # Main ApaleoAPIClient\n\u251c\u2500\u2500 constants.py # API endpoints and constants\n\u251c\u2500\u2500 exceptions.py # Custom exception classes\n\u251c\u2500\u2500 logging.py # Logging configuration\n\u251c\u2500\u2500 schemas.py # Pydantic schema definitions\n\u251c\u2500\u2500 typing.py # Type definitions\n\u2502\n\u251c\u2500\u2500 apaleo/ # API domain modules\n\u2502 \u251c\u2500\u2500 core/ # Core API (properties, inventory)\n\u2502 \u251c\u2500\u2500 identity/ # Identity & authentication\n\u2502 \u251c\u2500\u2500 payment/ # Payment processing\n\u2502 \u251c\u2500\u2500 webhook/ # Webhook management\n\u2502 \u251c\u2500\u2500 integration/ # Third-party integrations\n\u2502 \u251c\u2500\u2500 fiscalization/ # Tax and compliance\n\u2502 \u2514\u2500\u2500 profile/ # User profiles\n\u2502\n\u251c\u2500\u2500 http/ # HTTP layer\n\u2502 \u251c\u2500\u2500 auth.py # OAuth2 authentication\n\u2502 \u2514\u2500\u2500 transport.py # HTTP transport layer\n\u2502\n\u2514\u2500\u2500 services/ # Support services\n \u251c\u2500\u2500 list_fetcher.py # Pagination handling\n \u251c\u2500\u2500 response_handler.py # Response processing\n \u2514\u2500\u2500 response_validator.py # Response validation\n\n\n## \ud83d\uddfa\ufe0f Development Workflow\n\n### Available Tasks (via invoke)\n\nbash\ninv help # Show all available tasks\n\n# Development\ninv test # Run tests with pytest\ninv test-nox # Test across Python 3.10-3.13\ninv test-unit # Run unit tests only\ninv test-integration # Run integration tests only\n\n# Code Quality\ninv lint # Run all linters (ruff, mypy, etc.)\ninv format # Format code (ruff format, isort)\ninv check # Check code quality without fixing\n\n# Documentation \ninv docs # Build and serve documentation\ninv docs-build # Build documentation only\ninv docs-clean # Clean generated docs\n\n# Building\ninv build # Build package\ninv clean # Clean build artifacts\n\n\n### Testing Workflow\n\n#### Unit Tests\nbash\n# Run all unit tests\npytest tests/ -m unit\n\n# Run specific test file\npytest tests/test_client.py\n\n# Run with coverage\npytest tests/ --cov=apaleoapi --cov-report=html\n\n\n#### Integration Tests\nbash\n# Run integration tests (requires API credentials)\nAPALEO_CLIENT_ID=test APALEO_CLIENT_SECRET=test pytest tests/ -m integration\n\n\n#### Test Categories\nWe use pytest markers to categorize tests:\n\n- @pytest.mark.unit - Fast, isolated unit tests\n- @pytest.mark.integration - Tests that call real APIs\n- @pytest.mark.contract - API contract validation tests\n\n### Code Style\n\nWe use these tools for code quality:\n\n- ruff - Fast Python linter and formatter\n- mypy - Static type checking\n- isort - Import sorting\n- pytest - Testing framework\n\n#### Code Style Rules\n- Line length: 100 characters\n- Use type hints for all public APIs\n- Follow PEP 8 and Black formatting\n- Docstrings for all public functions/classes\n- Use dataclasses for structured data\n\n## \ud83d\udd25 Adding New Features\n\n### Adding a New API Domain\n\n1. New features, backwards compatible\n- PATCH: Bug fixes, backwards compatible\n\nbash\n# Update version in pyproject.toml\npoetry version patch|minor|major\n\n# Update CHANGELOG.md\n# Git commit and tag\ngit add .\ngit commit -m \"chore: bump version to vX.Y.Z\"\ngit tag vX.Y.Z\ngit push origin main --tags\n\n\n### Release Checklist\n- [ ] All tests pass across Python 3.10-3.13\n- [ ] Documentation is up to date\n- [ ] CHANGELOG.md is updated\n- [ ] Version is bumped in pyproject.toml\n- [ ] Git tag matches package version\n- [ ] GitHub release is created\n- [ ] Package is published to PyPI\n\n## \ud83d\ude80 Advanced Contributing\n\n### Performance Contributions\n\n- Use asyncio profiling for async code\n- Benchmark before/after changes\n- Consider memory usage for large datasets\n- Test with various network conditions\n\n### Security Contributions\n\n- Never log sensitive information (credentials, tokens)\n- Use environment variables for test credentials\n- Follow OWASP guidelines for HTTP clients\n- Consider timing attacks in authentication\n\n### API Design Principles\n\n- Favor async methods for I/O operations\n- Use type hints for all public APIs\n- Make required parameters explicit\n- Provide sensible defaults for optional parameters\n- Use dataclasses for complex parameters\n- Handle errors gracefully with typed exceptions\n\n## \ud83d\udcc9 Commit Message Format\n\nWe follow Conventional Commits:\n\n\n<type>([scope]): <description>\n\n[optional body]\n\n[optional footer(s)]\n\n\nTypes:\n- feat: A new feature\n- fix: A bug fix\n- docs: Documentation changes\n- style: Code style changes (formatting, etc.)\n- refactor: Code changes that neither fix bugs nor add features\n- test: Adding or updating tests\n- chore: Maintenance tasks\n\nExamples:\n\nfeat(core): add property search functionality\n\nfix(auth): handle token refresh race condition\n\ndocs: update installation guide for Poetry users\n\ntest(integration): add tests for pagination edge cases\n\n\n---\n\n## \ud83d\udc99 Thank You!\n\nEvery contribution helps make Apaleo API Client better for everyone.
